## 6. 팀별 피드백

### 기획팀

1. **권한 매트릭스 문서화 필요**: `canFollow`, `canVoteMvp` 등 역할별 허용/차단 기능을 기획서에 명시적 매트릭스로 정리해야 구현-QA 누수가 줄어듭니다. 현재 FollowButton 권한 누수가 대표적 사례입니다.
2. **역할 세분화 결정 필요**: 코치/스카우터를 DB에서 구분할 것인지, 단일 `"other"`로 유지할 것인지 확정해야 합니다. 현재 `permissions.ts`에 죽은 코드가 존재합니다.
3. **이모지 리액션 스펙 재확인**: 5종 vs 1종(응원만), 퀘스트 위치(홈 vs 프로필), 헤더 아이콘 구성(2개 vs 3개) 등 기획-구현 간 불일치가 여러 곳 있습니다.

### UI/UX팀

1. **바텀탭 깜빡임(FOUC) 해결**: role 로딩 전 5탭이 보였다가 3탭으로 전환되는 현상은 skeleton 또는 로딩 상태로 대체해야 합니다. 학부모 사용 경험에 큰 영향을 줍니다.
2. **부모 PDF 내보내기 진입점 설계**: 현재 부모가 자녀 프로필 PDF를 내보낼 방법이 없습니다. 자녀 대시보드나 공개 프로필에 진입점을 추가해야 합니다.
3. **스카우터 검색 필터 UX 설계**: 포지션/지역 필터는 스카우터의 핵심 탐색 도구이므로, 검색 오버레이에 필터 칩 또는 어드밴스드 검색 UI가 필요합니다.

### 개발팀

1. **`canFollow` 즉시 적용**: `permissions.ts`에 이미 함수가 있으니 FollowButton 컴포넌트와 `/api/follows` POST 핸들러에 role 체크만 추가하면 됩니다. 가장 빠른 핫픽스입니다.
2. **Parent Dashboard API 500 에러 디버깅**: 서버 로그 확인 후 원인 파악 필요. 자녀 연결 데이터 조회 쿼리에서 문제가 발생하는 것으로 추정됩니다.
3. **라우트 가드 일괄 점검**: 현재 바텀탭에서 숨기는 것으로 접근을 제한하고 있으나, URL 직접 입력 시 접근 가능합니다. `proxy.ts` 또는 layout 레벨에서 역할별 라우트 보호를 강화해야 합니다.
